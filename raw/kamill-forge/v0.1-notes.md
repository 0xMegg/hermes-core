# Kamill Forge v0.1 Source Notes

Status: Source record for Phase 0 documentation and policy.
Collected: 2026-05-13

This file records the user-approved source decisions for Kamill Forge v0.1. It is source material, not active policy. Active guidance belongs in `policy/kamill-forge.md`; synthesized durable knowledge belongs in `wiki/pages/kamill-forge-overview.md`.

## Naming Decision

The user-approved name for the Hermes Experience Distillation Loop / governance layer is **Kamill Forge**.

For Phase 0, `hermes-core` remains the repository and operating-layer template name. Kamill Forge names a governance layer over Hermes' existing self-improvement mechanisms, not a replacement repository or parallel project identity.

## Scope

Kamill Forge is not a parallel automatic learning system. It is a governance layer over Hermes' existing self-improvement mechanisms:

- memory
- skills
- session_search
- curator

## Approved v0.1 Defaults

- Threshold-primary design.
- Script-only watchdog for quiet-day checks.
- Avoid false positives.
- No LLM calls on quiet days.
- Unanchored Discord candidates stay global/unanchored until the user retargets them.
- Codex proposes.
- Claude judges.
- Kamill/Hermes synthesizes in Korean.
- User decides.
- Staged proposal only before explicit commit approval.
- No automatic memory, skill, Core, or project modifications without explicit approval.
- User-originated Kamill/Hermes improvement ideas are included in the same flow as a distinct Kamill Improvement Lane.

## Phase 0 Boundary

Phase 0 is a paper-run introduction of the governance layer into this repository. It may document the design, policy boundaries, provenance, and verification expectations. It must not introduce automatic execution scaffolding.

Out of scope for Phase 0:

- Repository rename.
- Broad Hermes-to-Kamill renaming.
- Watchdog scripts.
- Cron jobs.
- Skills.
- Hooks.
- Curator changes.
- Automatic memory, skill, Core, or downstream project mutation.

## 2026-05-13 Workflow Role Clarification

The user clarified that the active Kamill Forge workflow should use this model-role split:

- Opus plans first.
- Codex checks the plan and discusses it with Opus.
- Kamill/Hermes synthesizes the planning result in Korean.
- User decides whether implementation may proceed.
- Opus leads implementation after explicit approval.
- Codex owns development review, verification review, and closeout.
- Kamill/Hermes synthesizes the final result in Korean.

This clarification supersedes the earlier shorthand "Codex proposes / Claude judges" wording for active Kamill Forge workflow. The active policy and synthesized overview live in `policy/kamill-forge.md` and `wiki/pages/kamill-forge-overview.md`.

## 2026-05-14 v0.1 Completion Clarification

The user clarified that, excluding natural improvement as more real observations accumulate, Kamill Forge v0.1 should be treated as complete once the manual safe loop is reflected in workflow documentation and the repository is clean.

The completed v0.1 loop is:

1. Repeated experience, user-originated improvement ideas, or operating-layer observations may become candidates.
2. Quiet-day checks remain script-only, silent, and non-mutating.
3. Above-threshold explicit candidates may be normalized into dry-run ledger records.
4. A single explicit candidate may be rendered into a staged proposal draft.
5. Memory, skills, Core, project files, curator behavior, cron jobs, hooks, and downstream propagation remain blocked until the user approves the exact staged change.

Completion does not authorize automatic session scanning, scheduled execution, memory or skill writes, curator integration, project mutation, or downstream propagation. Those remain future scoped phases requiring separate approval.
